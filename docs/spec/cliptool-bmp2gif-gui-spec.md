# ClipTool 多图合成 GIF 交互升级需求规格

## 1. 背景

当前 `clip_bmp2gif` 工具已经从 Python demo 重构为 Go 可执行程序，核心能力是监听 Windows 剪贴板中的图片文件列表，将多张 BMP/PNG/JPG/RAW 指纹图合成为 GIF，并把 GIF 写回剪贴板，用户随后可直接 `Ctrl+V` 粘贴到文档或聊天工具。

现有问题是：用户在资源管理器中多选图片并 `Ctrl+C` 后，工具获得的剪贴板文件顺序不一定符合用户预期，导致生成 GIF 的帧顺序偶发错误。用户希望在不改变核心使用习惯的前提下，能够看到已复制图片、追加图片、去重、拖拽调整顺序，并配置 GIF 输出参数。

本规格用于记录原始需求、交互边界和首轮设计约束，避免后续迭代偏离真实使用场景。

## 2. 原始用户场景

目标用户不是开发者，也不会在终端输入命令。

原始操作习惯必须保留：

1. 用户在 Windows 文件资源管理器中选择一张或多张图片。
2. 用户按 `Ctrl+C`。
3. 工具从剪贴板读取图片文件。
4. 工具在窗口中追加显示图片缩略图。
5. 用户可以继续在资源管理器中选择图片并 `Ctrl+C`，工具继续去重追加。
6. 用户在工具窗口内拖拽缩略图调整 GIF 帧顺序。
7. 用户点击“生成”按钮或按 Enter。
8. 工具按当前顺序和参数合成 GIF。
9. 工具把 GIF 写回剪贴板。
10. 工具清空当前图片列表。
11. 用户在目标软件中按 `Ctrl+V` 粘贴 GIF。

关键结论：CLI 不是主交互入口。GUI 不是炫技目标，而是为了解决“可视化排序、参数调整、明确生成时机”这三个问题。

## 3. 要解决的问题

### 3.1 帧顺序可控

用户需要在生成前看到所有帧，并能通过拖拽调整顺序。

### 3.2 多次复制可追加

用户可能不是一次选齐所有图片，而是分多次从资源管理器复制。工具需要把新的图片追加到当前列表，不应覆盖已有列表。

### 3.3 去重

同一张图片被重复复制时，工具应避免重复添加。默认去重策略建议使用文件路径归一化；后续如确有需要再引入内容 hash。

### 3.4 明确生成时机

现有程序检测到 2 张及以上图片后自动生成。升级后应改为用户显式触发生成，避免用户还没整理完顺序就生成。

### 3.5 参数可调

用户希望在生成前调整 GIF 输出参数。首轮只做必要参数，避免过度设计。

## 4. 不解决的问题

以下内容不进入首轮 MVP：

1. 不做完整图片编辑器。
2. 不做裁剪、标注、画笔、文字编辑。
3. 不做时间轴级别的复杂动画编辑。
4. 不做项目文件保存和历史记录。
5. 不做云同步。
6. 不做多平台适配，优先 Windows。
7. 不做复杂主题系统。
8. 不强求替代通用 GIF 制作软件。
9. 不改变用户在资源管理器中 `Ctrl+C`、目标软件中 `Ctrl+V` 的核心流程。

## 5. 现成项目调研结论

已搜索 GitHub 类似项目，未发现能直接替代本工具业务流的成熟项目。

可参考项目：

1. `cavazzatommaso/Looply`
   - 类型：Tauri + Svelte 图片序列转 GIF 工具。
   - 可参考：拖拽图片、参数面板、预览、现代 UI。
   - 不适合直接替代：不处理 Windows 剪贴板监听、去重追加、生成后写回剪贴板，也不支持当前工具的 RAW/BMP 指纹图处理逻辑。

2. `faith289/GIF-maker`
   - 类型：PyQt6 图片序列转 GIF 工具。
   - 可参考：拖拽重排、延迟、裁剪、抖动、锐化、量化等参数设计。
   - 不适合直接替代：Python/PyQt 分发较重，且没有贴合剪贴板工作流。

结论：可借鉴交互，不建议直接替换。当前工具的差异化价值在“资源管理器复制图片 -> 工具累积排序 -> GIF 写回剪贴板”的窄流程。

## 6. GUI 技术选型

推荐方案：Wails + Svelte 或 Wails + Vanilla TypeScript。

选择理由：

1. 保留现有 Go 后端，复用图片加载、GIF 编码和 Windows 剪贴板逻辑。
2. 前端实现缩略图网格、拖拽排序、按钮、滑块、参数面板更高效。
3. Windows 下复用 WebView2，体积和资源占用低于 Electron。
4. 比 Fyne/Gio 更容易做出可接受的现代 UI。
5. 比 Tauri 更适合当前 Go 代码，不需要把后端迁移到 Rust。

备选判断：

| 方案 | 判断 |
| --- | --- |
| Wails | 推荐，平衡开发效率、UI 质量和 Go 复用 |
| Fyne | 纯 Go，但拖拽排序和精致 UI 成本较高 |
| Gio | 轻量但偏底层，开发成本高 |
| Tauri | UI 能力强，但后端 Rust 化，不适合当前代码 |
| PyQt6 | 功能好做，但分发重 |
| Electron | 开发快但占用过重 |

## 7. 首轮 MVP 范围

### 7.1 必须实现

1. 启动后显示一个轻量窗口，不再以终端后台为主要入口。
2. 监听剪贴板文件列表。
3. 支持多次 `Ctrl+C` 追加图片。
4. 支持按路径去重。
5. 显示图片缩略图。
6. 支持缩略图拖拽排序。
7. 支持删除单张图片。
8. 支持清空当前列表。
9. 支持 GIF 帧延迟参数。
10. 支持 Enter 或“生成”按钮触发合成。
11. 合成成功后把 GIF 写回剪贴板。
12. 合成成功后清空当前列表。
13. 合成失败时保留当前列表，并显示错误。

### 7.2 可选但适合第一版一起做

1. 显示当前帧数。
2. 显示当前状态：监听中、已追加、生成中、已复制到剪贴板、错误。
3. 支持自然文件名排序按钮，用于快速恢复常见顺序。
4. 支持反转顺序按钮。
5. 支持窗口置顶开关。

### 7.3 后续迭代

1. GIF 宽高缩放。
2. 预览 GIF 动画。
3. 布局模式：仅动图、缩略图对照 + 当前帧。
4. 色板和量化策略。
5. 保存到文件。
6. 全局快捷键。
7. 托盘常驻。
8. 内容 hash 去重。
9. 最近参数记忆。

## 8. 核心交互设计

窗口布局建议：

```text
--------------------------------------------------+
| ClipTool 多图合成 GIF                  [置顶]   |
+--------------------------------------------------+
|                                                  |
|  空状态：复制图片后会显示在这里                  |
|                                                  |
|  [1] [2] [3] [4] [5] ...                         |
|  缩略图可拖拽排序，可选中删除                    |
|                                                  |
+--------------------------------------------------+
| 帧数: 5   延迟: [ 500 ms ]   排序: [自然] [反转] |
| [清空] [删除选中]                 [生成 GIF]     |
+--------------------------------------------------+
| 状态：已生成 GIF，可直接粘贴                     |
+--------------------------------------------------+
```

交互规则：

1. 复制新图片时追加到列表末尾。
2. 同一路径已存在时跳过，不改变原有顺序。
3. 拖拽排序只影响当前列表，不修改文件。
4. 删除只从当前列表移除，不删除磁盘文件。
5. 生成按钮在有效图片少于 2 张时禁用。
6. Enter 等同于点击生成。
7. 生成期间禁用生成按钮，避免重复触发。
8. 生成失败不清空列表。
9. 生成成功后清空列表，进入下一轮采集。

## 9. 架构边界

建议拆分为四层：

```text
clipboard/
  watcher.go       // Windows 剪贴板监听，产出文件路径列表

core/
  image_loader.go  // 图片加载，复用现有 BMP/PNG/JPG/RAW 支持
  gif_encoder.go   // GIF 编码和参数
  frame_store.go   // 追加、去重、删除、重排
  options.go       // 输出参数

app/
  bindings.go      // 暴露给 Wails 前端的方法和事件

frontend/
  thumbnail grid   // 缩略图列表
  sortable frames  // 拖拽排序
  controls         // 参数与按钮
```

原则：

1. 剪贴板读取只负责发现文件，不负责排序和编码。
2. 图片加载只负责把文件转为 `image.Image`，不关心 UI。
3. `FrameStore` 是当前会话的唯一帧顺序来源。
4. GIF 编码只接收有序帧和参数。
5. 前端只表达用户操作，不直接处理图片二进制。

## 10. 数据模型草案

```go
type FrameItem struct {
    ID       string
    Path     string
    Name     string
    ThumbURL string
    Width    int
    Height   int
}

type GifOptions struct {
    DelayMS int
    Layout  string
}

type FrameStore struct {
    Frames []FrameItem
}
```

`ID` 建议首轮使用规范化路径 hash。后续如果要允许同一路径重复作为不同帧，再调整为内容 hash + append 序号。

## 11. 剪贴板监听策略

首轮建议沿用现有轮询模型，而不是立即切换到 Windows 消息监听。

理由：

1. 当前 Go 代码已验证轮询可用。
2. 轮询实现简单，迁移风险低。
3. MVP 的核心风险在 GUI 状态和排序交互，不在剪贴板事件模型。

后续如需要降低资源占用，再考虑 `AddClipboardFormatListener` + `WM_CLIPBOARDUPDATE`。

## 12. 参数范围

首轮参数保持克制：

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| 帧延迟 | 500ms | 现有 `gifDelay = 50` 对应 500ms |
| 布局 | 对照布局 | 沿用当前“左侧缩略图 + 右侧当前帧”逻辑 |
| 最少帧数 | 2 | 少于 2 张不生成 |

暂不做复杂质量参数。当前已有 256 色固定调色板，先复用。

## 13. 错误处理

| 场景 | 行为 |
| --- | --- |
| 剪贴板没有文件 | 忽略 |
| 剪贴板文件不是支持图片 | 忽略并显示状态提示 |
| 部分图片加载失败 | 跳过失败项，保留成功项 |
| 有效图片少于 2 张 | 禁用生成 |
| GIF 编码失败 | 显示错误，保留列表 |
| 写入剪贴板失败 | 显示错误，保留列表 |
| 用户重复复制同一批图片 | 全部去重跳过 |

## 14. 验证标准

MVP 完成时至少验证：

1. 一次复制多张 BMP 图片，窗口出现缩略图。
2. 多次复制图片，窗口追加且不覆盖。
3. 重复复制同一图片，不重复添加。
4. 拖拽缩略图后，生成 GIF 顺序与 UI 顺序一致。
5. 删除单张后，生成 GIF 不包含该图片。
6. 清空后，列表为空。
7. 少于 2 张时不能生成。
8. 生成成功后，目标软件可 `Ctrl+V` 粘贴 GIF。
9. 生成成功后列表自动清空。
10. 生成失败时列表保留。

## 15. 关键决策

1. 不继续把 CLI 当作主交互方案。
2. GUI 的目标是服务原始复制粘贴工作流，不是做通用 GIF 编辑器。
3. 首轮保留 Go 核心能力，优先引入 Wails 做轻量 GUI。
4. 首轮不重写 GIF 编码算法。
5. 首轮不引入复杂配置系统。
6. 首轮不追求全平台，优先 Windows。

## 16. 待确认问题

1. 缩略图区域是否需要固定窗口置顶。
2. 生成后是否一定清空，还是提供“生成后保留列表”选项。
3. 默认布局是否继续使用当前“缩略图对照 + 当前帧”。
4. 是否需要支持把当前 GIF 同时保存到 `temp/` 供排查。
5. 是否保留旧版无 GUI 监听模式作为兼容入口。
